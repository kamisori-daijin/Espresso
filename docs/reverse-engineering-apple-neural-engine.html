Reverse-Engineering Apple's Neural Engine for Direct Inference

Apple's Neural Engine is one of the most powerful ML accelerators in consumer hardware. CoreML gives you access to about 20% of its performance. Here's how we found the other 80%.

Note on private APIs: Everything described here uses undocumented, private Apple APIs. This works today on macOS 15 / Apple Silicon. It could break in future OS updates. We monitor changes and will adapt, but use this in production at your own risk.

The Stack

Before going further, here's what's actually running when you execute a model on Apple Silicon:

Your Code (Swift/Python/Obj-C) ↓ CoreML Framework ← Where most developers stop ↓ MIL Compiler (espresso, private) ← Compiles to ANE E5 binary ↓ _ANEClient (private) ← Espresso enters here ↓ _ANEInMemoryModel (private) ↓ ANE Driver (kernel space) ↓ Neural Engine Silicon ← 18 TFLOPS fp16 on M3 Max

CoreML is a convenient wrapper around the private ANE stack. It adds model validation, format conversion, compile caching, and a clean API. It also adds latency — marshaling, per-token overhead, and abstraction layers that aren't free.

Espresso calls _ANEClient and _ANEInMemoryModel directly, bypassing CoreML entirely.

How We Found the Private APIs

1. class-dump and otool

The starting point is class-dump on CoreML's private framework and the ANE daemon:

# Dump the ANE private framework headers
class-dump /System/Library/PrivateFrameworks/ANECompilerService.framework/

# Check what CoreML links against
otool -L /System/Library/Frameworks/CoreML.framework/Versions/A/CoreML | grep ANE

# Look at what symbols are exported
nm /System/Library/PrivateFrameworks/ANECompilerService.framework/Versions/A/ANECompilerService \
  | grep -i "ANEClient\|ANEModel\|ANERequest"

This revealed the main classes: _ANEClient, _ANEInMemoryModel, _ANERequest, and _ANESharedEvents.

2. Hopper / Binary Ninja for call graphs

With class names in hand, we used Hopper to trace the call graph from CoreML's MLModel.prediction down to the ANE dispatch. Key discovery: CoreML calls [_ANEClient evaluateWithModel:request:] and that's the last stop before the kernel.

3. Dynamic instrumentation with LLDB

With the call graph mapped, we used LLDB to intercept ANE calls and inspect arguments:

(lldb) breakpoint set --name "-[_ANEClient evaluateWithModel:request:]"
(lldb) po $arg2  # Print the _ANEInMemoryModel object
(lldb) po $arg3  # Print the _ANERequest object

This let us see exactly what a valid _ANEInMemoryModel looks like — what fields it expects, how weights are laid out, what the IOSurface format should be.

The dlopen Bridge

Direct Objective-C calls to private APIs would require a framework link, which Apple would reject. Espresso uses dlopen to load the private framework at runtime and dlsym to resolve symbols:

// ANEInterop.m — the ObjC/C bridge layer
#include <dlfcn.h>

static void* ane_handle = NULL;
static Class _ANEClientClass = nil;
static Class _ANEInMemoryModelClass = nil;

void ane_interop_init(void) {
    ane_handle = dlopen(
        "/System/Library/PrivateFrameworks/ANECompilerService.framework/"
        "Versions/A/ANECompilerService",
        RTLD_NOW | RTLD_LOCAL
    );
    if (!ane_handle) { return; }

    _ANEClientClass = NSClassFromString(@"_ANEClient");
    _ANEInMemoryModelClass = NSClassFromString(@"_ANEInMemoryModel");
}

// Compile a MIL program to ANE binary
id ane_interop_compile_model(NSString* milText, NSArray* weightBlobs, ...) {
    if (!_ANEInMemoryModelClass) { ane_interop_init(); }

    id model = [_ANEInMemoryModelClass alloc];
    // ... (initialization from reversed header)
    return model;
}

The Swift layer in Espresso calls into this ObjC bridge cleanly:

// Swift side — ANERuntime module
import ANEInterop

let handle = ane_interop_compile_model(
    milText as NSString,
    weightBlobs as NSArray,
    inputSizes, outputSizes
)
// handle is an opaque ANE model reference

MIL: The Intermediate Representation

MIL (Multi-Level Intermediate Representation) is Apple's internal IR for neural network programs. It's a text format — the same one CoreML writes internally before handing it to the ANE compiler. Espresso writes MIL directly.

The format looks like this:

program(1.3) {
  func main(
    x: tensor<fp16, [1, 768, 1, 32]>,
    w_q: const tensor<fp16, [1, 768, 1, 768]>
  ) -> (tensor<fp16, [1, 768, 1, 32]>) {
    block0():
      // Layer norm (RMSNorm: square, reduce_sum, pow for rsqrt)
      sq = mul(x=x, y=x)
      sum = reduce_sum(x=sq, axes=[1], keep_dims=true)
      inv_std = pow(x=sum, y=fp16(-0.5))  // rsqrt via pow

      // Linear projection via conv (ANE's native operation)
      q = conv(
        x=x_normed,
        weight=w_q,
        pad_type="valid",
        strides=[1,1], pad=[0,0,0,0],
        dilations=[1,1], groups=1
      )
      return (output)
  }
}

MIL uses a tensor layout that matches the ANE's native IOSurface format: [N, C, H, W] with channel-first ordering. Tokens are packed along the width dimension. Weights are compiled into the E5 binary at compile time — weight swapping at runtime is not supported.

What We Learned the Hard Way: MIL Gotchas

Not everything that looks like valid MIL actually compiles on the ANE. We hit every one of these walls:

BROKEN rsqrt operator

rsqrt(x=tensor) exists in the CoreML Python SDK but is not available in raw MIL text format for the ANE compiler. Use pow(x=tensor, y=fp16(-0.5)) instead — identical result, actually compiles.

BROKEN reduce_mean operator

reduce_mean does not exist in the ANE MIL dialect. Use reduce_sum followed by a manual divide. We found this by trying to write LayerNorm and getting InvalidMILProgram with no useful error message.

BROKEN softmax on non-power-of-2 dimensions

Softmax over a non-power-of-2 dimension size compiles but returns InvalidMILProgram at eval time. Attention score dimensions must be powers of 2. This cost us days before we isolated the cause.

BROKEN INT8 / quantized weights

Both constexpr_affine_dequantize and cast(uint8→fp16) produce InvalidMILProgram. The private MIL compiler does not support non-fp16 weight types. fp16 only.

BLOCKED slice_by_index on function inputs with complex programs

Slicing a function input combined with RMSNorm + convolutions + attention produces InvalidMILProgram at compile time. Workaround: pre-shape your inputs before passing them to the ANE program — let the caller slice, not the MIL program.

WORKS Everything else we needed

conv, matmul, softmax (power-of-2 dims), sigmoid, mul, add, reshape, transpose, reduce_sum, reduce_max, pow, concat (12+ inputs) — all confirmed working. Enough to build full transformer attention and FFN blocks.

IOSurface: The Memory Model

The ANE doesn't operate on regular heap memory. It operates on IOSurface buffers — shared memory objects managed by the OS that can be accessed by the CPU, GPU, and ANE without copying.

Espresso creates input IOSurfaces, writes fp16 data to them using NEON-vectorized C functions, passes them to the ANE via _ANERequest, and reads output IOSurfaces directly after eval:

// Create an IOSurface for ANE input
let surface = IOSurfaceCreate([
    kIOSurfaceWidth: width,
    kIOSurfaceHeight: height,
    kIOSurfaceBytesPerElement: 2,  // fp16
    kIOSurfacePixelFormat: kCVPixelFormatType_OneComponent16Half
] as CFDictionary)

// Write data (NEON-vectorized C function in ANEInterop)
ane_interop_write_fp32_to_fp16_surface(surface, floatData, count)

// After eval(), read output directly — no copy
let tokenId = ane_interop_io_argmax_fp16_spatial_slice(
    outputSurface, vocabSize: 32000
)

The zero-copy design means ANE output bytes are read directly from the mapped IOSurface — the same memory the ANE wrote to. No intermediate buffer, no format conversion, no extra allocation.

What Didn't Work (Dead Ends)

_ANEVirtualClient

We found a class called _ANEVirtualClient with 22 instance methods and 66 class methods, including what looked like async dispatch and pipeline control. Exciting, until every instantiation path returned nil — classic IOKit entitlement gate. This API requires kernel-level entitlements that only Apple's own processes have.

Async eval via _ANESharedEvents

We tried to pipeline ANE dispatches using _ANESharedEvents backed by MTLSharedEvent. The standard eval path hung on hardware before the completion callback fired. Metal-based event signaling doesn't bridge to the ANE scheduler the way it does to the GPU scheduler.

Metal/ANE hybrid attention

We prototyped a split path: ANE for QKV projection, Metal for the attention math, ANE for FFN. Metal attention over 12 heads measured 0.31 ms/eval. But inserting Metal into the dependency chain destroyed any overlap opportunity — FFN layer N depends on Metal layer N, QKV layer N+1 depends on FFN layer N. Net result: 1.8 ms/token slower than pure ANE.

Multi-layer cache channel packing

We tried packing two layers' KV caches onto the channel axis to fuse two decode steps into one dispatch. Both the full-pack and reduced-pack variants produced InvalidMILProgram at compile time on M3 Max macOS 15. The MIL compiler doesn't handle large channel-packed inputs with complex graphs.

The Architecture That Works

After exhausting the above dead ends, the architecture that actually hit the performance targets:

  1. RWKV-style recurrent stepping — fixed-size state vectors instead of growing KV caches. Constant latency at any context length.
  2. 3-layer kernel fusion — compile 3 transformer layers into one MIL program. 6 layers = 2 ANE dispatches total.
  3. Fused RMSNorm + classifier head — keep the output head on ANE with lane-packed layout for compatibility.
  4. Direct-select argmax — NEON scan on IOSurface memory. No logits copy to CPU.

Lessons for Anyone Going Further

Start with small programs. A single-op identity kernel (input → output, no processing) is your baseline. If that works, add one op at a time. InvalidMILProgram often comes from op combinations, not individual ops.

Dimension alignment matters. The ANE is optimized for 64-byte-aligned shapes. Keep channel dimensions as multiples of 64 where possible. Smaller widths (< 32) on trunk kernels reliably produce statusType=0x9 at eval time.

Weights are baked in at compile time. There's no runtime weight swapping. If you need to run multiple models, compile separate ANE binaries. The OS caches them in ~/Library/Caches/, so cold compile happens once per distinct program.

Compilation is expensive the first time. Our first cold compile of a 6-layer recurrent model took 131 seconds. Subsequent runs (after OS cache warmup) take 350ms. Plan your iteration cycle accordingly — don't change MIL structure in a tight loop.

IOSurface format is non-negotiable. [1, C, 1, S] with kCVPixelFormatType_OneComponent16Half. Any deviation breaks the contract. The channel axis is how the ANE identifies tensor type; the spatial axis packs sequence positions.
