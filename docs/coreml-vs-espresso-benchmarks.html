CoreML vs Espresso: Benchmarks on Apple Silicon

We benchmarked CoreML's neural engine path against Espresso's direct ANE inference on Apple Silicon. The gap is 4.76x. Here's what we measured, how we measured it, and why the gap exists.

Bottom line: On an M3 Max, Espresso generates tokens at 519 tok/s (1.929 ms/token). CoreML's .cpuAndNeuralEngine path runs at 152 tok/s (6.582 ms/token) on the same model. That's a 3.41x difference in end-to-end generation, and 4.76x at the raw inference layer.

The Setup

Both frameworks ran the same 6-layer, 768-dim transformer (stories110M architecture, Llama-style). Hardware: Apple M3 Max with 16-core ANE, macOS 15.0, 64GB unified memory.

CoreML used the .cpuAndNeuralEngine compute units — Apple's recommended path for ANE acceleration. Espresso compiled MIL programs directly to ANE E5 binaries through the private _ANEClient API, bypassing CoreML entirely.

Methodology: 20 warmup iterations, then 100 timed iterations. Median reported. Both ran back-to-back on the same machine with no other compute-intensive processes.

Why the Gap Exists

1. CoreML adds abstraction layers

CoreML is a high-level API. Under .cpuAndNeuralEngine, your model goes through the CoreML compiler, the MIL IR, and multiple optimization passes before it hits the ANE. That compilation happens once per session, but the dispatch overhead — serializing inputs, marshaling across the CPU/ANE boundary, deserializing outputs — happens every token.

2. Espresso compiles MIL directly

Espresso generates MIL programs as strings and calls the private _ANEClient API directly, bypassing CoreML's entire abstraction stack. The compiled E5 binary is cached by the OS; subsequent calls are pure IOSurface buffer swaps and ANE dispatches.

// Espresso: direct MIL compilation
let kernel = try ANEKernel(
    milText: myMIL,           // Raw MIL program string
    weights: weightBlobs,     // fp16 weight blobs
    inputSizes: [inputShape],
    outputSizes: [outputShape]
)
try kernel.eval()             // Direct ANE dispatch, no CoreML

3. Fused multi-layer kernels

The 6-layer model normally requires 12 separate ANE dispatches (one for attention, one for FFN per layer). Espresso fuses every 3 layers into a single MIL program — so the full 6-layer trunk runs in 2 ANE dispatches. Each dispatch boundary is dead latency: kernel launch overhead, IOSurface sync, and scheduler scheduling. Fewer dispatches = faster generation.

4. Zero-copy I/O

Espresso reads ANE outputs via NEON-vectorized C functions operating directly on IOSurface memory — no copy, no format conversion. The argmax for token selection runs as a NEON/vDSP scan over the mapped surface. CoreML copies outputs to a CPU-accessible buffer before any host code can touch them.

Reproducing These Numbers

# Clone and run the benchmark
git clone https://github.com/christopherkarani/Espresso.git
cd Espresso

# Run the full inference benchmark (requires Apple Silicon, macOS 15+)
swift run espresso-bench --ane-only --inference --layers 6 --warmup 20 --iterations 100

# Compare against CoreML baseline
./espresso compare --no-power "Hello world"

The --diagnostic flag prints a breakdown of trunk vs. head vs. dispatch overhead:

swift run espresso-bench --ane-only --inference --diagnostic --layers 6

What About M1 and M2?

The benchmarks above are on M3 Max (16-core ANE, 18 TFLOPS fp16). The performance gap between CoreML and direct ANE access is structural, not hardware-specific — it comes from the abstraction layers, not the silicon. You should expect similar relative speedups on M1 and M2 chips, with absolute throughput scaling roughly with ANE TFLOPS.

Caveats

Espresso uses private APIs (_ANEClient, _ANEInMemoryModel). These are reverse-engineered and undocumented. They work today on macOS 15 / Apple Silicon but could break in future OS updates. If API stability is a hard requirement, CoreML is the right choice. If you're building a performance-critical research tool or framework and can tolerate that risk, Espresso is significantly faster.

Espresso also only supports macOS 15+ on Apple Silicon (M1 or newer). CoreML runs on Intel Macs, iOS, and older OS versions. Different use case.
