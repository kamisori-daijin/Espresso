Why Espresso

Everything CoreML costs you, eliminated.

CoreML adds four compounding overhead layers at every decode step. Espresso removes all of them.

🧊
Compile once, decode forever
MIL programs compile to ANE E5 binaries once and are reused across every decode step. No per-token recompilation. No sequence-length-triggered rebuilds.
⚡
Fused 3-layer kernels
6 transformer layers run in 2 ANE dispatches, not 6. Each fused kernel runs RMSNorm → Attention → FFN → residual for 3 layers in a single graph. ~0.57ms overhead eliminated.
🔄
Zero-copy IOSurface I/O
KV cache lives in IOSurface buffers the entire decode loop — no marshaling, no Array allocation, no float32 conversion. NEON-vectorized reads, vDSP argmax.
🧩
Direct ANE access
Uses _ANEClient and _ANEInMemoryModel — the same private APIs CoreML uses internally — via dlopen. Bypasses the entire CoreML scheduler stack.
🎓
Training on ANE
Full forward + backward passes with gradient accumulation and Adam optimizer running directly on the Neural Engine. CoreML doesn't support this.
📦
Zero dependencies
Swift 6.2, Foundation, and Accelerate. That's it. No CoreML dependency. No Python toolchain. One swift build.
How it works

The CoreML layer, removed.

Espresso speaks MIL directly to the ANE — no intermediate framework.

1

Generate MIL programs

MILGenerator produces Metal Intermediate Language text for each kernel variant — fused attention+FFN, decode KV-cache, classifier head. 28+ kernel variants cover the full transformer architecture.

2

Compile to ANE E5 binary via _ANEClient

ANEInterop's ObjC bridge dlopens Apple's private ANEFramework and calls _ANEClient.compile() directly. The result is an ANE E5 binary cached by the system — compile once, run thousands of times.

3

Bind inputs and weights to IOSurface

Input activations, KV caches, and weight tensors live in IOSurface buffers — GPU/ANE-accessible shared memory. No copy on dispatch. No marshaling on output.

4

Eval + zero-copy read

_ANEInMemoryModel.eval() runs the compiled graph. Results are read directly from the output IOSurface using NEON-vectorized code and vDSP argmax — no intermediate buffer.

Architecture

Layered, composable, zero-dep.

ANEInterop (ObjC/C — private API bridge) └── ANETypes (~Copyable tensors, IOSurface I/O) ├── MILGenerator (28+ kernel variants) │ └── ANERuntime (compile, eval, surfaces) │ └── Espresso (generation, decode, training) │ ├── EspressoTrain (CLI) │ └── EspressoBench (CLI) └── CPUOps (Accelerate/vDSP kernels) └── Espresso
Quick start

One command to see it live.

Requires Apple Silicon (M1+) and macOS 15. Swift 6 comes with Xcode 16.

$ git clone https://github.com/christopherkarani/Espresso
$ cd Espresso && ./espresso
# Builds Swift targets, downloads GPT-2 assets, launches TUI
# Side-by-side: Espresso vs CoreML, live token generation
 
$ swift run espresso-bench --decode --decode-steps 32 --layers 6
# Raw ANE decode benchmark — reproduces the 1.08 ms/token number
 
$ ./espresso doctor
# Diagnose your environment if ./espresso doesn't start cleanly

Minimal usage
import ANERuntime import ANETypes // Compile once — reuse for thousands of decode steps let kernel = try ANEKernel( milText: generator.milText, weights: weightBlobs, inputSizes: [inputSize], outputSizes: [outputSize] ) // Decode step — zero CoreML try kernel.eval() // Zero-copy output from IOSurface let surface = kernel.outputSurface(at: 0) let nextToken = SurfaceIO.argmaxFP16SpatialSlice(surface, lane: 0)
