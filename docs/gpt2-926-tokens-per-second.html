Where Does 926 tok/s Come From?

926 tok/s is the peak inference rate measured on the trunk model — 6 transformer layers, ANE only, no logits head overhead. The all-in generation number (including the output head) is 519 tok/s on the same hardware. Here's the breakdown of what we optimized to get there.

Step 1: Bypass CoreML entirely

CoreML adds abstraction overhead — it's designed for portability, not speed. Espresso calls the private _ANEClient API directly, writing MIL (Multi-Level Intermediate Representation) programs as text strings and compiling them to ANE E5 binaries:

import ANERuntime

// Compile a MIL program directly to ANE binary
let kernel = try ANEKernel(
    milText: milProgram,     // Raw MIL text (see below)
    weights: weightBlobs,   // [Data] of fp16 weight tensors
    inputSizes: [...],
    outputSizes: [...]
)

// Each eval() is a direct IOSurface dispatch to ANE
try kernel.eval()

The MIL program is a text string — Espresso generates it programmatically from weight shapes. It looks like this:

program(1.3) {
  func main(x: tensor, ...) -> (tensor) {
    block0():
      // RMSNorm
      sq = mul(x=x, y=x)
      ss = reduce_sum(x=sq, axes=[1], keep_dims=true)
      // ... attention, FFN ...
      return (xNext)
  }
}

Step 2: Fuse 3 transformer layers into 1 ANE dispatch

A standard 6-layer transformer requires 12 ANE dispatches per token (attention + FFN per layer). Each dispatch carries kernel launch overhead. Espresso fuses every 3 layers into a single MIL program — 6 layers become 2 dispatches:

Before fusion: 6 layers × 2 kernels = 12 ANE dispatches per token
After fusion: 2 fused kernels (3 layers each) = 2 ANE dispatches per token
Dispatch overhead saved: ~0.95 ms/token (10 × 0.095ms per dispatch) 
// RWKVStyleFusedThreeLayerKernelSet: 3 layers in one dispatch
let triplet0 = try RWKVStyleFusedThreeLayerKernelSet(layers: [layer0, layer1, layer2])
let triplet1 = try RWKVStyleFusedThreeLayerKernelSet(layers: [layer3, layer4, layer5])

// Full 6-layer trunk = 2 eval() calls
try triplet0.eval(x: input, stateIn: state0)
try triplet1.eval(x: triplet0.output, stateIn: state1)

Step 3: Switch to a recurrent (RWKV-style) architecture

Attention transformers scale as O(context²) — each new token attends to every previous token. At inference time, this means growing KV caches and increasing per-token latency as context grows. Espresso's recurrent architecture maintains a fixed-size state vector instead, so latency stays constant regardless of how many tokens you've generated:

ArchitectureContext 32Context 256Context 1024Context 4096
Transformer (attention) 2.85 ms/tok 2.87 ms/tok 2.91 ms/tok ~3.1 ms/tok
Recurrent (RWKV-style) 0.21 ms/tok 0.22 ms/tok 0.21 ms/tok 0.22 ms/tok

Recurrent inference stays flat at ~0.22 ms/tok even at 4,096 context length. The transformer path grows slowly (it's still fast because of fused kernels and IOSurface caching), but the recurrent path wins decisively at scale.

Step 4: Direct-select argmax on IOSurface

The naïve output head: copy ANE output to CPU → run softmax over 32k vocab → select top token. That's ~1.35 ms/token just for logits.

Espresso instead reads the ANE output surface directly via NEON-vectorized C functions and runs a vDSP argmax scan on the raw fp16 data — no full copy, no softmax, no format conversion:

// NEON scan over IOSurface memory (no copy)
let tokenId = SurfaceIO.argmaxFP16SpatialSlice(
    surface: kernel.outputSurface(at: 0),
    vocabSize: 32000
)

This cut output head latency from 1.35 ms/token to 0.94 ms/token — a 30% reduction just from eliminating the copy.

The Full Generation Pipeline

Step 1 · 0.05 ms
Token Embedding Lookup

Convert token ID to fp16 embedding vector (768-dim). CPU op via Accelerate.

Step 2 · 0.43 ms
Fused Triplet 0 (Layers 0–2)

3 recurrent transformer layers in a single ANE dispatch. Input: [1,768,1,32], Output: [1,768,1,32], state updated in-place.

Step 3 · 0.46 ms
Fused Triplet 1 (Layers 3–5)

3 more layers, same pattern. Total trunk: 2 ANE dispatches, ~0.89 ms.

Step 4 · 0.94 ms
Fused RMSNorm + Classifier Head

ANE-side final norm + linear projection to vocab (32k). Lane-packed for ANE compatibility. Direct argmax on output IOSurface.

Total: 1.929 ms/token (519 tok/s) all-in including embedding + 6-layer trunk + output head + token selection. Trunk alone: 0.893 ms/token (1,120 tok/s).

Using Espresso in Your Project

// Package.swift
dependencies: [
    .package(url: "https://github.com/christopherkarani/Espresso.git", from: "1.1.0")
],
targets: [
    .target(name: "MyApp", dependencies: [
        .product(name: "ANERuntime", package: "Espresso"),
        .product(name: "Espresso", package: "Espresso")
    ])
]
import Espresso

// Load the full generation pipeline
let model = try await ANEDirectGenerationModel.load(
    config: ModelConfig.default,  // 768-dim, 12-layer
    weightsPath: URL(fileURLWithPath: "path/to/weights")
)

// Autoregressive generation
var tokens: [Int] = tokenizer.encode("Once upon a time")
for _ in 0..<200 {
    let next = try model.nextToken(tokens: tokens)
    tokens.append(next)
    print(tokenizer.decode([next]), terminator: "")
}

Caveats and Limitations

Espresso uses private APIs. They work on macOS 15 / Apple Silicon today, but Apple could change them. The project monitors OS updates and adapts as needed.

The 926 tok/s figure is trunk-only (no output head). All-in with generation is 519 tok/s. Both are real measured numbers — we just report both because the distinction matters for understanding where time is spent.

This is a 6-layer demo model (stories110M, ~110M parameters). Full GPT-2 (12 layers) and larger models will be slower, scaling roughly linearly with layer count. Expect ~260 tok/s for full GPT-2 with the current architecture.
